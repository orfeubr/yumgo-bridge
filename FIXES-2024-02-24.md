# Correções Implementadas - 24/02/2026

## 🔧 Problema 1: Domínios Criados com UUID

**Erro**: Ao criar novo tenant no painel, domínios eram criados com UUID ao invés do slug amigável.

**Exemplo**:
- ❌ Antes: `144c5973-f985-4309-8f9a-c404dd11feae.yumgo.com.br`
- ✅ Depois: `marmitariadagi.yumgo.com.br`

**Arquivo Corrigido**: `/app/Filament/Resources/TenantResource/Pages/CreateTenant.php:19`

```php
// ❌ ANTES
$slug = $tenant->id; // O ID do tenant é o slug

// ✅ DEPOIS
$slug = $tenant->slug; // 🔥 CORRIGIDO: Usar slug ao invés do UUID
```

**Impacto**: Agora quando criar um novo restaurante no painel central, os domínios serão automaticamente criados com o slug amigável que você definir no formulário.

## 🔧 Problema 2: OAuth Login Falhando (Google/Facebook)

**Erro**:
```
SQLSTATE[23502]: Not null violation: 7 ERROR: null value in column "phone"
of relation "customers" violates not-null constraint
```

**Causa**: Campo `phone` era obrigatório (NOT NULL), mas login social não fornece telefone.

**Arquivos Corrigidos**:

1. **Migration**: `/database/migrations/2026_02_24_001007_create_central_customers_table.php:19`
```php
// ❌ ANTES
$table->string('phone', 20)->unique();

// ✅ DEPOIS
$table->string('phone', 20)->unique()->nullable(); // 🔥 NULLABLE para OAuth login
```

2. **Banco de Dados**: Executado diretamente via SQL
```sql
ALTER TABLE customers ALTER COLUMN phone DROP NOT NULL;
```

**Verificação**:
```bash
php artisan tinker --execute="
\$result = DB::select(\"SELECT is_nullable FROM information_schema.columns
WHERE table_name = 'customers' AND column_name = 'phone'\");
echo 'Phone nullable: ' . \$result[0]->is_nullable;
"
# Resultado: Phone nullable: YES ✅
```

**Impacto**: Login social (Google/Facebook) agora funciona corretamente. Telefone será solicitado posteriormente via WhatsApp.
